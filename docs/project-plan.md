# Project Plan: Ethereum Validator Deposit Data Generator (`eth-deposit-gen`)

## Summary

A Go CLI that generates Launchpad-compatible `deposit_data-<ts>.json` files
from existing BLS pubkeys and an EIP-2335 keystore. The work is organized into
four phases mirroring the PRD milestones (M1–M4). Phase 1 stands up the full
correctness-critical pipeline on Hoodi end-to-end. Phase 2 adds mainnet
constants and the byte-for-byte cross-check against `staking-deposit-cli`
fixtures. Phase 3 layers on P1 polish (parallelism, dry-run, structured logs,
optional verify step). Phase 4 hardens the release path. The critical risk —
producing a deposit that locks funds — is bought down in Phase 1 by golden-
file tests against the official CLI, so the remaining phases compose lower-
risk increments around an already-correct core.

Estimated total wall-clock: **~16–22 dev-days for one developer** (4–5 weeks
with normal review cadence), with meaningful parallelism opportunities inside
Phase 1 that can compress this to ~3 weeks.

## Prerequisites

- Repo access; Go ≥ 1.22 (workspace pins 1.23) installed.
- CGO toolchain on the dev host (required by `herumi/bls-eth-go-binary`):
  clang/gcc on macOS/Linux.
- A throwaway BLS keystore + passphrase + matching pubkey generated by
  `staking-deposit-cli` (Hoodi), checked into `testdata/` for golden-file
  tests. Ditto for mainnet (separate fixture).
- A locally-installed `staking-deposit-cli` (Python) for producing golden
  fixtures and (P1) optional shell-out verification.
- `go.work` updated to register `./cmd/eth-deposit-gen` (per architecture.md).
- Decisions confirmed before coding starts:
  - Withdrawal-credentials policy for v1 (hard-coded address vs. flag).
  - Pinned `deposit_cli_version` string (e.g. `"2.7.0"`).

## Phase 1 — M1: Core (Hoodi)

**Goal:** End-to-end Hoodi deposit generation. Operator can run the CLI
against a Hoodi keystore and pubkey set and get a byte-for-byte
`staking-deposit-cli`-equivalent JSON file. All P0 functional requirements
satisfied for Hoodi.

**Duration estimate:** Large — **8–11 dev-days**.

**Depends on:** Prerequisites only.

### Tasks

- [ ] **1.1 — Module + workspace bootstrap** *(small, 0.5d)*
  - Create `go/cmd/eth-deposit-gen/` Go module; register in `go/go.work`.
  - Add baseline `Makefile` targets (`test`, `lint`, `build`).
  - Wire `go vet` + `staticcheck` + `go mod verify` into CI skeleton.
  - Dependencies: none. Complexity: low.

- [ ] **1.2 — `internal/network` package** *(small, 0.5–1d)*
  - Implement `Network` enum, `Params`, `Lookup`, `ParseFlag`.
  - Hard-code Hoodi `GENESIS_FORK_VERSION = 0x10000910`; leave mainnet
    placeholder returning an explicit "not yet enabled" error if selected.
  - Define `DomainDeposit = 0x03000000` and `ZeroGenesisValidatorsRoot`.
  - Unit tests: byte-for-byte assertions, `ParseFlag` rejection cases.
  - Dependencies: 1.1. Complexity: low.

- [ ] **1.3 — `internal/ssz` hand-rolled hash_tree_root** *(medium, 1.5–2d)*
  - Implement `uint64Chunk`, `padRight`, `merkleize`.
  - Implement `HashTreeRoot` for `DepositMessage`, `DepositData`,
    `ForkData`, `SigningData` per chunk tables in
    `docs/research/bls-ssz-libraries.md`.
  - Implement `ComputeDomain` and `ComputeSigningRoot`.
  - Unit tests against consensus-spec test vectors (publicly available).
  - Fuzz `merkleize` and `uint64Chunk` for crash safety.
  - Dependencies: 1.1. Complexity: medium. **CRITICAL PATH.**

- [ ] **1.4 — `internal/bls` herumi wrapper** *(medium, 1–1.5d)*
  - One-time `Init` + `SetETHmode(EthModeDraft07)` with idempotent guard.
  - `NewSigner(secret []byte) (Signer, error)`; expose `Sign` + `PublicKey`.
  - `DefaultVerifier() Verifier`.
  - All boundary types are `[N]byte`.
  - Round-trip unit tests + ETH ciphersuite byte-vector assertion.
  - CGO build verified locally on darwin/arm64 + linux/amd64.
  - Dependencies: 1.1. Complexity: medium. **CRITICAL PATH.**

- [ ] **1.5 — `internal/keystore` EIP-2335 loader** *(medium, 1.5d)*
  - Integrate `wealdtech/go-eth2-wallet-encryptor-keystorev4`.
  - `KeyLoader.Load(ctx, path, PassphraseSource) (Key, error)`.
  - Implement `NewEnvSource` (env var name) and `NewTermPromptSource`
    (`golang.org/x/term`).
  - Explicit `Key.Zeroize()`; passphrase byte slice zeroized after use.
  - Reject `version != 4`; clear typed errors for missing/malformed/wrong-pw.
  - Tests: scrypt + PBKDF2 fixtures, wrong-pw path, missing-file path,
    bad-version path, `bytes`-source for non-TTY CI.
  - Dependencies: 1.1. Complexity: medium. *(Parallelizable with 1.3/1.4.)*

- [ ] **1.6 — `internal/deposit` orchestrator** *(medium, 1.5–2d)*
  - Define `Request`, `Entry`, `Generator`.
  - Implement the 10-step per-pubkey pipeline (architecture.md §deposit).
  - Per-entry signer-pubkey equality check; self-verify after sign; recompute
    `deposit_data_root` before emit.
  - Sentinel errors: `ErrPubkeyMismatch`, `ErrSelfVerifyFailed`.
  - Unit tests with fake `Signer`/`Verifier` covering mismatch + verify-fail
    + success paths; real `ssz` package used in tests.
  - Dependencies: 1.2, 1.3, 1.4. Complexity: medium. **CRITICAL PATH.**

- [ ] **1.7 — `internal/output` JSON writer** *(small–medium, 1d)*
  - `jsonEntry` struct with exact field order matching official CLI.
  - Lowercase hex no-0x for byte fields; `amount` as JSON number.
  - Atomic write: `.tmp` + fsync + rename.
  - sha256 of final file returned to caller.
  - `NewDryRunWriter(io.Writer)` stubbed (full P1 polish in Phase 3 but
    in-memory writer needed for golden tests now).
  - Tests: byte-for-byte fixture; atomic-rename under `t.TempDir`.
  - Dependencies: 1.6 (for `deposit.Entry` shape). Complexity: low–medium.

- [ ] **1.8 — `internal/cli` flag parsing** *(small–medium, 1d)*
  - `urfave/cli/v2` app exposing the four required flags plus
    `--passphrase-env`.
  - Pubkey hex validation (48 bytes, valid compressed G1 — `bls` decode
    check); reject 0x-prefixed *or* unprefixed inconsistencies.
  - `--output-dir` must exist & be writable.
  - Confirmation banner: network + first/last pubkey to stderr before
    signing (PRD risk mitigation).
  - Fuzz the pubkey parser.
  - Dependencies: 1.2. Complexity: low–medium.

- [ ] **1.9 — `cmd/eth-deposit-gen` wiring** *(small, 0.5d)*
  - Compose loader → signer → generator → writer per architecture.md §main.
  - Exit-code mapping (`errors.Is` → 2/3/4) via `exitCodeFor`.
  - `defer key.Zeroize()` immediately after Load.
  - Final stderr summary (path, sha256, count, network).
  - Dependencies: 1.5, 1.6, 1.7, 1.8. Complexity: low.

- [ ] **1.10 — Hoodi golden-file integration test** *(medium, 1.5–2d)*
  - Produce `testdata/hoodi/{keystore.json, passphrase.txt, pubkeys.txt,
    deposit_data-expected.json}` via `staking-deposit-cli`.
  - End-to-end test loads keystore, runs generator, serializes via dry-run
    writer, diffs against expected JSON field-by-field (ignoring filename
    timestamp).
  - Document `make refresh-golden` regeneration ritual + which upstream CLI
    version produced the fixtures.
  - Dependencies: 1.9. Complexity: medium. **GATING CHECK FOR M1 EXIT.**

### Parallelizable inside Phase 1

After 1.1+1.2, tasks 1.3, 1.4, 1.5 can be developed in parallel by separate
developers / sessions. 1.7 and 1.8 can begin once 1.6's types are stubbed.

### Top 3 Risks (Phase 1)

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Hand-rolled SSZ diverges from spec (wrong chunk order, wrong endian) | Critical — wrong signatures across all entries | Medium | Unit tests against consensus-spec vectors *and* golden-file diff vs. official CLI in 1.10; do not advance to M2 until 1.10 is green. |
| CGO/herumi build issues on dev host (esp. Apple Silicon, alpine CI) | High — blocks all signing work | Medium | Validate `bls.Init()` round-trip on every dev machine + CI runner in 1.4 before downstream work depends on it. |
| Withdrawal-credentials policy unresolved | Medium — blocks 1.6 emit shape | Medium | Resolve in prerequisites; if still open, ship hard-coded address from a constant and a clearly-marked TODO; no flag change required to revisit. |

### Phase Exit Criteria

- `go test ./...` green, including 1.10 golden-file diff.
- `eth-deposit-gen --network hoodi ...` produces a JSON file that diffs
  identically to the official CLI's output for the same inputs.
- Self-verification, atomic write, exit codes, and `defer Zeroize` all
  exercised in tests.
- One end-to-end manual run completed against a real Hoodi keystore on the
  developer's machine; output uploaded to the Hoodi Launchpad and accepted.

## Phase 2 — M2: Mainnet enablement

**Goal:** Unlock mainnet by adding its constants and proving correctness with
the same golden-file gate used for Hoodi. End-to-end Hoodi deposit
submitted on-chain to validate real-world acceptance.

**Duration estimate:** Small — **2–3 dev-days**.

**Depends on:** Phase 1 complete (especially 1.3, 1.6, 1.10).

### Tasks

- [ ] **2.1 — Mainnet constants in `internal/network`** *(small, 0.5d)*
  - Add `Mainnet` `Params{GenesisForkVersion: 0x00000000}`.
  - Update `Lookup`/`ParseFlag` to accept `mainnet`.
  - Unit-test byte values explicitly.
  - Dependencies: 1.2. Complexity: low.

- [ ] **2.2 — Mainnet golden-file fixture + test** *(medium, 1d)*
  - Generate `testdata/mainnet/...` from `staking-deposit-cli`.
  - Mirror the Hoodi golden-file test for mainnet.
  - Confirmation banner test asserts "mainnet" is printed before signing.
  - Dependencies: 2.1, 1.10. Complexity: low–medium. **GATING CHECK FOR M2.**

- [ ] **2.3 — On-chain Hoodi deposit dry-run** *(small, 0.5–1d)*
  - Run real Hoodi deposit using output of `eth-deposit-gen`; confirm
    the deposit contract accepts it on first submission.
  - Record tx hash + validator pubkey in `docs/validation/hoodi-e2e.md`.
  - Dependencies: 2.1. Complexity: low. *(Non-blocking for code, gating for
    confidence to release.)*

- [ ] **2.4 — Network safety banner hardening** *(small, 0.5d)*
  - When `--network mainnet`, require an extra confirmation token (e.g.
    print banner + a `--i-understand-this-is-mainnet` style guard *or* a
    second-line confirmation if TTY) — per PRD risk register on accidental
    mainnet submission. (Implementation choice flagged for review.)
  - Dependencies: 2.1, 1.8. Complexity: low.

### Top 3 Risks (Phase 2)

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Hoodi on-chain submission rejected despite passing golden test | Critical — invalidates the whole correctness story | Low | 2.3 is mandatory gate before M4 release; if rejected, root-cause before any mainnet promotion. |
| Mainnet constants typo (`0x00000000` vs. wrong byte) | Critical — locks funds | Low | 2.2 golden file pins byte-for-byte; explicit byte-array unit test in 2.1. |
| Network-confusion UX too lenient | High — operator picks wrong network | Medium | 2.4 enforces explicit mainnet acknowledgement; banner test in 2.2. |

### Phase Exit Criteria

- Mainnet and Hoodi golden-file tests both green in CI.
- One Hoodi deposit accepted on-chain (tx hash recorded).
- Mainnet-only confirmation guard implemented and tested.

## Phase 3 — M3: P1 polish

**Goal:** Make the tool pleasant and safe for batch operators — parallel
signing, dry-run, structured logging, optional cross-check with the official
CLI.

**Duration estimate:** Medium — **3–4 dev-days**.

**Depends on:** Phase 2 complete (correctness story locked in).

### Tasks

- [ ] **3.1 — Parallel signing worker pool** *(medium, 1–1.5d)*
  - Add `--parallel N` flag (default = `runtime.NumCPU()` or 1 for
    determinism; finalize during review).
  - Implement bounded goroutine pool inside `deposit.Generator.Generate`.
    Preserve entry order (collect into indexed slice).
  - Self-verification still runs per entry; any failure cancels context.
  - Benchmark to confirm ≥ 200 sigs/sec NFR; document baseline.
  - Dependencies: 1.6. Complexity: medium.

- [ ] **3.2 — `--dry-run` mode** *(small, 0.5d)*
  - Flag toggles `output.NewDryRunWriter(os.Stdout)`; no file written.
  - Stderr summary still printed (sha256 of the would-be file).
  - Dependencies: 1.7, 1.8. Complexity: low.

- [ ] **3.3 — Structured logging with `log/slog`** *(small–medium, 1d)*
  - `--verbose` toggles `slog.LevelDebug`; `--json-logs` switches handler.
  - Logger threaded via `context` to packages that want to log (none on
    the signing path by design — log only at orchestration boundaries).
  - Audit: no log line emits passphrase, decrypted key, signing-root, or
    keystore contents; lint test enforces this via grep-style assertion.
  - Dependencies: 1.9. Complexity: low–medium.

- [ ] **3.4 — `--verify-with-deposit-cli` cross-check** *(medium, 1d)*
  - Optional post-step that shells out to the user's installed
    `staking-deposit-cli` verify routine on the produced JSON.
  - If the CLI isn't installed, emit a clear non-fatal warning unless the
    flag is set (in which case fail loudly).
  - Test using a stubbed `exec.Cmd` wrapper.
  - Dependencies: 1.9. Complexity: medium.

- [ ] **3.5 — Progress indicator for >5 entries** *(small, 0.5d)*
  - Minimal counter to stderr (avoid heavy TUI deps); suppressed under
    `--json-logs` or non-TTY stdout.
  - Dependencies: 1.6, 3.3. Complexity: low.

### Top 3 Risks (Phase 3)

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Parallelism breaks determinism of output order | Medium — golden tests flake | Medium | Indexed result slice + order assertion in golden test re-run under `--parallel`. |
| Logging accidentally leaks key bytes / passphrase | Critical — defeats security NFR | Low | 3.3 includes a grep-test that scans test output for known fixture secrets. |
| Shell-out to `staking-deposit-cli` introduces a flaky external dep | Medium — CI noise | Medium | 3.4 is opt-in only; stub the binary in tests; document required version. |

### Phase Exit Criteria

- All P1 flags implemented with tests.
- Benchmark documents ≥ 200 sigs/sec.
- Logging audit (lint test) green.
- Golden tests still pass under `--parallel >1`.

## Critical Path

The shortest sequence that gates the whole project:

```
1.1 (bootstrap)
  └─▶ 1.2 (network) ──▶ 1.3 (ssz)  ──┐
                       1.4 (bls)  ──┤
                       1.5 (keystore) ──┤
                                        └─▶ 1.6 (deposit) ──▶ 1.7 (output)
                                                            ──▶ 1.8 (cli)
                                                                  └─▶ 1.9 (main wire-up) ──▶ 1.10 (golden Hoodi) ★
                                                                                                └─▶ 2.1 (mainnet const) ──▶ 2.2 (golden mainnet) ★
                                                                                                                                  └─▶ 2.3 (on-chain Hoodi)
                                                                                                                                  └─▶ Phase 3 polish ──▶ Phase 4 release
```

**★ = correctness gate.** The two golden-file checks (1.10 Hoodi, 2.2
mainnet) are the only things standing between this tool and the "lose 32 ETH"
failure mode. Everything else can slip; those two cannot.

Tasks **1.3, 1.4, 1.6, 1.10, 2.2** are on the critical path. Of these, **1.3
(SSZ) and 1.6 (orchestrator)** carry the most correctness risk; budget review
time for them disproportionately.

## Parallel Work Streams

If more than one developer is available:

- **Stream A — Crypto core:** 1.3 (ssz) → contributes to 1.6.
- **Stream B — Crypto wrapper:** 1.4 (bls) → contributes to 1.6.
- **Stream C — I/O edges:** 1.5 (keystore) and 1.7 (output) → independent.
- **Stream D — CLI surface:** 1.8 (cli flags) once 1.2 stabilizes.

Streams A+B+C+D run concurrently after 1.1/1.2, converge at 1.6/1.9.

Within Phase 3, 3.1 (parallel signing), 3.3 (logging), and 3.4 (verify
cross-check) are mutually independent and can be parallelized.

## Technical Spikes / Open Questions

(Restated from PRD + architecture.md; resolve before or during Phase 1.)

- **Withdrawal-credentials for v1:** hard-coded operator address vs.
  `--withdrawal-address` flag. **Recommendation:** ship a flag in Phase 1
  with a sensible default if any non-trivial operator usage is expected;
  otherwise hard-code and add the flag in a v1.1.
- **`CLIVersion` pinning:** literal `const` vs. `-ldflags -X` build-time
  variable. **Recommendation:** start with a `const`, switch to ldflags only
  if release engineering asks for it.
- **`--verify-only` mode:** orchestrator already supports it. Decision:
  expose in v1 (small surface) or v1.1.
- **`--pubkeys @file` indirection:** PRD open question. Decision: defer to
  v1.1 unless an early user hits shell-length limits.
- **Mainnet confirmation guard form:** banner + explicit ack flag vs. banner
  + TTY prompt. Decide in 2.4.
- **CGO cross-compilation strategy for release:** zig-cc vs. per-arch
  runners. Decide in 4.1; prototype in a Phase 3 sidebar if risky.

## Decision Log

| # | Decision | Rationale | Source |
|---|----------|-----------|--------|
| D1 | Hand-roll SSZ for the 4 required structs | Smallest supply-chain surface on the signing path; only ~150 LOC | ADR-001 |
| D2 | Use `herumi/bls-eth-go-binary` | Ethereum-flavored API out of the box; battle-tested; perf is non-binding | ADR-002 |
| D3 | Modular monolith inside one Go module under `go/cmd/eth-deposit-gen` | Matches existing repo layout; keeps CGO isolated from future pure-Go libs | ADR-003 |
| D4 | Compile-time network constants only | Wrong fork version is unrecoverable; no runtime config or network fetch | ADR-004 |
| D5 | Verify-before-write with atomic file output | No partial output; abort on any self-verify failure | ADR-005 |
| D6 | Passphrase via env-var-name or TTY prompt only — never a flag value | Avoid shell history and `/proc/<pid>/cmdline` leakage | PRD Security NFR + keystore-decoding research |
| D7 | Golden-file diff vs. `staking-deposit-cli` is the primary correctness gate | Only way to credibly assure byte-for-byte Launchpad compatibility | PRD Risks; architecture.md Testing Strategy |
| D8 | Phase 1 targets Hoodi only; mainnet enabled in Phase 2 behind its own golden fixture | Reduce blast radius during initial development; force a deliberate mainnet enablement step | PRD Milestones M1/M2 |
