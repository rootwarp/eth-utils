# v1.0.0 Release Checklist

Runbook for cutting and verifying the `eth-deposit-gen` v1.0.0 release.
Execute each section in order. Record the date + initials in the sign-off table when done.

> **Scope of this document:** steps the *user* must execute. No automation
> pushes the tag or publishes the release — those are explicit human actions.
> The prior dry-run procedure is in `docs/validation/release-strategy.md`;
> this document covers only the steps that differ for a real stable release.

---

## 1. Pre-flight

Before touching `git`:

- [ ] Confirm `develop` is at the expected base commit:
  ```bash
  git -C /path/to/repo rev-parse develop
  # Expected: the commit that merged PR #24 (feat/issue-24-release-notes)
  ```
- [ ] Confirm `docs/validation/audit-v1.md` is present and signed off:
  ```bash
  grep "Sign-off" /path/to/repo/docs/validation/audit-v1.md
  # Must contain: "Second-engineer sign-off" line — should be filled in by PR reviewer
  ```
- [ ] Confirm all three release documents are merged to `develop`:
  ```bash
  git -C /path/to/repo show develop:RELEASE_NOTES_v1.0.0.md >/dev/null && echo OK
  git -C /path/to/repo show develop:CHANGELOG.md >/dev/null && echo OK
  git -C /path/to/repo show develop:docs/validation/v1.0.0-release-checklist.md >/dev/null && echo OK
  ```
- [ ] Confirm CI is green on `develop`:
  ```bash
  gh run list --branch develop --workflow eth-deposit-gen.yml --limit 3
  # All recent runs should show "completed / success"
  ```
- [ ] Confirm zero open P0 issues against v1.0.0:
  ```bash
  gh issue list --label "P0" --state open
  # Should return: no issues
  ```

## 2. Merge `develop` → `main`

Use fast-forward only to preserve the linear history.

```bash
# From the repository root
git checkout main
git pull --ff-only origin main
git merge --ff-only develop
# Expected: Fast-forward merge; no merge commit created.
git log --oneline -5
# The top commit should be the feat/issue-24-release-notes merge commit.
```

Do not push yet — verify the merge commit is correct before pushing.

```bash
git status
# Expected: nothing to commit, working tree clean
git diff origin/main..main --stat
# Expected: files added (RELEASE_NOTES_v1.0.0.md, CHANGELOG.md, docs/validation/v1.0.0-release-checklist.md, audit-v1.md)
```

Push main:

```bash
git push origin main
```

---

## 3. Tag and push

Create an annotated tag pointing at the merge commit on `main`.

```bash
# Verify HEAD on main is correct before tagging
git rev-parse HEAD
git log --oneline -1
# Confirm this is the commit you intend to tag.

# Create the annotated tag
git tag -a v1.0.0 -m "eth-deposit-gen v1.0.0

First stable release. Supports Hoodi and mainnet (with safety gate).
See RELEASE_NOTES_v1.0.0.md for full details."

# Verify the tag was created
git tag --list "v1.0.0"
# Expected: v1.0.0

git show v1.0.0 --no-patch
# Expected: tag object with annotation, pointing at HEAD on main.
```

Push the tag to trigger the release CI:

```bash
git push origin v1.0.0
```

Push the subdirectory-prefixed tag to enable `go install` resolution (does NOT
re-trigger the release workflow — the workflow triggers on `v*` only):

```bash
git tag go/v1.0.0 v1.0.0^{}
git push origin go/v1.0.0
```

Monitor the release workflow:

```bash
gh run list --workflow=release.yml --limit=5
gh run watch
# Select the run triggered by the v1.0.0 tag push.
# All three test jobs (linux/amd64, linux/arm64, darwin/arm64) must pass
# before the goreleaser job runs.
```

## 4. Verify GitHub release artifacts

After the `goreleaser` job completes:

```bash
gh release view v1.0.0
```

Check each item:

- [ ] Release is published as **stable** (NOT pre-release, NOT draft).
  - `goreleaser.yaml` sets `prerelease: auto`; `v1.0.0` has no pre-release
    semver suffix, so goreleaser publishes it as a stable release automatically.
  - Confirm: `gh release view v1.0.0 --json prerelease --jq '.prerelease'`
    — expected output: `false`
- [ ] 4 archives present:
  - `eth-deposit-gen_darwin_amd64.tar.gz`
  - `eth-deposit-gen_darwin_arm64.tar.gz`
  - `eth-deposit-gen_linux_amd64.tar.gz`
  - `eth-deposit-gen_linux_arm64.tar.gz`
- [ ] `checksums.txt` present (SHA-256 for all 4 archives)
- [ ] 4 SBOM files present (attached by `anchore/sbom-action` in the goreleaser job):
  - `eth-deposit-gen_darwin_amd64.sbom.spdx.json`
  - `eth-deposit-gen_darwin_arm64.sbom.spdx.json`
  - `eth-deposit-gen_linux_amd64.sbom.spdx.json`
  - `eth-deposit-gen_linux_arm64.sbom.spdx.json`
- [ ] Replace the auto-generated release notes body with the curated draft.
  goreleaser's `changelog:` block auto-generates the body from commit messages —
  it does NOT pull from `RELEASE_NOTES_v1.0.0.md`. Run:
  ```bash
  gh release edit v1.0.0 --notes-file RELEASE_NOTES_v1.0.0.md
  ```

  Then reload the release page and confirm the body matches `RELEASE_NOTES_v1.0.0.md`.
- [ ] Links in the release notes resolve:
  - `docs/architecture.md` — exists in repo
  - `docs/validation/audit-v1.md` — exists in repo
  - `CHANGELOG.md` — exists in repo

List all assets to confirm:
```bash
gh release view v1.0.0 --json assets --jq '[.assets[].name] | sort | .[]'
```

Expected output (9 assets):

## 5. Fresh-VM smoke tests (user-executed)

These must be run on real machines (not the build host) to validate that the
release archives are self-contained and runnable.

For each platform, download the archive, verify the checksum, extract, and run
`--help` and `--version`.

### darwin/arm64 (Apple Silicon Mac)

```bash
# Download archive and checksums
gh release download v1.0.0 \
  --pattern 'eth-deposit-gen_darwin_arm64.tar.gz' \
  --pattern 'checksums.txt' \
  --dir /tmp/smoke-darwin-arm64

cd /tmp/smoke-darwin-arm64

# Verify checksum
grep 'eth-deposit-gen_darwin_arm64.tar.gz' checksums.txt | shasum -a 256 -c
# Expected: eth-deposit-gen_darwin_arm64.tar.gz: OK

# Extract and test
tar -xzf eth-deposit-gen_darwin_arm64.tar.gz
./eth-deposit-gen --version
# Expected: eth-deposit-gen version 1.0.0 (commit=<sha>, built=<rfc3339>)

./eth-deposit-gen --help
# Expected: exits 0, prints usage with all flags listed
```

### linux/amd64

```bash
# On a linux/amd64 VM (or Docker on matching host):
gh release download v1.0.0 \
  --pattern 'eth-deposit-gen_linux_amd64.tar.gz' \
  --pattern 'checksums.txt' \
  --dir /tmp/smoke-linux-amd64

cd /tmp/smoke-linux-amd64

grep 'eth-deposit-gen_linux_amd64.tar.gz' checksums.txt | sha256sum -c
# Expected: eth-deposit-gen_linux_amd64.tar.gz: OK

tar -xzf eth-deposit-gen_linux_amd64.tar.gz
./eth-deposit-gen --version
# Expected: eth-deposit-gen version 1.0.0 (commit=<sha>, built=<rfc3339>)

./eth-deposit-gen --help
# Expected: exits 0, prints usage
```

Note: the linux binaries are statically linked against musl libc (built via
`goreleaser-cross` container with GNU cross-compilers). They should run without
additional shared library install steps on any modern Linux distribution.

### darwin/amd64 and linux/arm64

Follow the same pattern, substituting the appropriate archive name:


## 6. `go install @v1.0.0` smoke test

> **Subdirectory module tag note:** The Go module is at `go/go.mod` with path
> `github.com/rootwarp/eth-utils/go`. Per Go's subdirectory module convention,
> the GOPROXY-resolvable version tag is `go/v1.0.0`, not `v1.0.0`. The goreleaser
> release CI is triggered by `v*` tags (so the GitHub release is `v1.0.0`), but
> `go install` resolves versions via the subdirectory-prefixed tag.
>
> **Before this step, push the subdirectory-prefixed tag:**
> ```bash
> git tag go/v1.0.0 v1.0.0^{}  # tag the same commit as v1.0.0
> git push origin go/v1.0.0
> ```
> This tag does NOT trigger the release workflow (the workflow triggers on `v*`,
> and `go/v1.0.0` starts with `go/`, not `v`) — it only enables `go install` resolution.

On a machine with a CGO toolchain (gcc or clang) and Go 1.25+:

```bash
CGO_ENABLED=1 go install github.com/rootwarp/eth-utils/go/cmd/eth-deposit-gen@v1.0.0
$(go env GOPATH)/bin/eth-deposit-gen --version
```

Expected output:
```
eth-deposit-gen version 1.0.0 (commit=<full-commit-sha>, built=<rfc3339-timestamp>)
```

Note: the version printed is `1.0.0` without a `v` prefix. This is intentional:
goreleaser's `{{.Version}}` template strips the leading `v` from the semver tag,
and that value is injected via `-X main.version={{.Version}}` in the ldflags.

## 7. Hoodi dry-run smoke test

Run the full pipeline in dry-run mode against the committed fixture. The output
must match `go/testdata/hoodi/deposit_data-expected.json` field-for-field.

```bash

export HOODI_PASS=hoodi-golden-test-passphrase

eth-deposit-gen \
  --network hoodi \
  --dry-run \
  --keystore-dir go/testdata/hoodi/keystores \
  --pubkeys 8420760d0de00ed65f290ab2122e65933e168539ad261b5e444a5094c649272527a1509dd105a801922c359e46e33fb9 \
  --output-dir /tmp \
  --passphrase-env HOODI_PASS \
  > /tmp/hoodi-smoke.json
```

Compare against the expected fixture (field-by-field, ignoring the filename
timestamp which is not part of the JSON body):

```bash
# Compare all JSON fields except deposit_cli_version (pinned to 2.7.0 in both)
python3 -c "
import json, sys
got = json.load(open('/tmp/hoodi-smoke.json'))

assert got == exp, f'Mismatch:\ngot={json.dumps(got, indent=2)}\nexp={json.dumps(exp, indent=2)}'
```
