# Hoodi Golden-File Validation

## Summary

The Hoodi golden-file integration test (`TestHoodiGoldenDeposit` in
`test/e2e/hoodi_test.go`) is the M1 exit gate for Phase 1 of `eth-deposit-gen`.
It proves end-to-end correctness of the signing pipeline on the Hoodi testnet:
keystore load, BLS sign, SSZ hash, and JSON serialization all produce the
same byte sequence as the committed fixture.

## Fixture provenance

## Important: self-consistency test (Phase 1)

The golden fixtures were generated programmatically from a fixed 32-byte BLS
secret (`goldenSecret` in `hoodi_test.go`) using the Go implementation itself.
**This proves internal self-consistency of the pipeline, not byte-for-byte
compatibility with the Python `staking-deposit-cli` v2.7.0.**

`staking-deposit-cli` v2.7.0 was not available on the build system at Phase 1.
The `deposit` binary present at `/Users/nil/opt/bin/deposit` is v1.2.2, which
uses a different workflow (mnemonic-based key generation) and cannot be used to
generate fixtures from a chosen secret.

## How to refresh fixtures

If the signing pipeline, SSZ implementation, or output format changes, refresh
the golden fixtures by running:

```bash
make refresh-golden
# or equivalently:
REFRESH_GOLDEN=1 CGO_ENABLED=1 go test -run TestRefreshHoodiGolden -v -timeout 120s ./test/e2e/
```

Then commit the updated `testdata/hoodi/` directory and update this document.

## CI integration

`TestHoodiGoldenDeposit` runs unconditionally as part of `go test ./...`. It is
**not** skipped in CI. The test decrypts `keystore.json` (scrypt N=262144,
~325 ms on modern hardware) on every invocation.
